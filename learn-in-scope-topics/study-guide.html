Domain 1 — Agentic Architecture & Orchestration (27%)

1. Agentic loop implementation D1

What it is

The control-flow pattern that lets Claude repeatedly call tools until a task is complete. You inspect response.stop_reason on every response and decide whether to loop or terminate.

STOP — Send request · Test stop_reason · Operate tools · Proceed (loop) or Terminate

Key facts

  • stop_reason == "tool_use" → execute all tool_use blocks, append tool_result blocks, loop.
  • stop_reason == "end_turn" → task complete, extract final text.
  • Tool results are appended as {"role": "user", "content": [{"type": "tool_result", "tool_use_id": ..., "content": ...}]}.
  • Tool results MUST be in conversation history so the model reasons from them next turn.

Anti-patterns (all explicitly called out in the guide)

Do NOT parse assistant text for words like "DONE" or "complete" to terminate the loop.
Do NOT use a fixed iteration cap as the PRIMARY stopping mechanism (a small safety cap is fine as a backstop).
Do NOT check for the presence of assistant text as a completion indicator — Claude may emit text mid-tool-use.

Minimal correct pattern

while True:
    resp = client.messages.create(model="claude-sonnet-4-6", tools=TOOLS, messages=messages)
    if resp.stop_reason == "end_turn":
        return extract_text(resp)
    if resp.stop_reason == "tool_use":
        messages.append({"role": "assistant", "content": resp.content})
        tool_results = execute_all_tool_uses(resp.content)
        messages.append({"role": "user", "content": tool_results})

2. Multi-agent orchestration D1

What it is

Coordinator-subagent architecture: one coordinator agent decomposes a task, delegates to specialised subagents, aggregates their results, and decides whether more work is needed.

HUB — Hub-and-spoke (all routing through coordinator) · Unique context per subagent · Breadth check on decomposition

Key facts

  • Hub-and-spoke: subagents don't talk to each other; the coordinator owns all inter-agent communication, error handling, and info routing.
  • Subagents operate with isolated context — they don't inherit the coordinator's history automatically.
  • The coordinator dynamically selects subagents based on query complexity — it doesn't always run the full pipeline.
  • Iterative refinement: coordinator evaluates synthesis output for gaps, re-delegates with targeted queries, re-invokes synthesis.

The exam trap

If a multi-agent system produces incomplete output but every subagent "completed successfully," suspect overly narrow decomposition by the coordinator. Example from the guide: query "creative industries" gets decomposed into only "visual arts" subtasks; music, writing, and film are silently omitted. The subagents worked correctly within their assigned scope — the problem is the scope.

Design principles

  • Partition scope across subagents to minimize duplication (distinct subtopics or source types per agent).
  • Route ALL subagent communication through the coordinator for observability and consistent error handling.
  • Loop refinement until coverage is sufficient — don't stop after one synthesis pass.

3. Subagent context management D1

What it is

Managing the state that subagents need — explicitly passing context in, persisting state so subagents can recover from crashes, keeping subagent context isolated from the main conversation.

PIC — Pass context explicitly · Isolate per subagent · Crash-recover via manifest

Key facts

  • Explicit context passing: everything the subagent needs must be in its invocation prompt.
  • Structured state persistence: write a manifest file recording completed steps.
  • Crash recovery: on restart, read the manifest and skip work that's already done.
  • Isolation prevents subagent output from polluting the main conversation context.

Manifest pattern

{
  "task_id": "research-2026-07-02",
  "coordinator_state": {"phase": "synthesis", "gaps_found": []},
  "subagent_results": {
    "search_agent": {"status": "done", "findings_file": "findings.json"},
    "analysis_agent": {"status": "in_progress", "started_at": "..."}
  }
}

Domain 2 — Tool Design & MCP Integration (18%)

4. Tool interface design D2

What it is

Designing the tool schema, description, and naming so Claude reliably picks the right tool for the right job.

SLAT — Split semantic operations · Limit tool count · Action-name clearly · Tell what it returns

Key facts

  • 4-5 tools per agent, not 18. Too many tools measurably degrades selection reliability.
  • Agents given tools outside their specialisation misuse them (a synthesis agent shouldn't have web_search).
  • Scoped tool access per role. Provide narrow cross-role tools only for high-frequency needs (e.g. verify_fact on a synthesis agent).
  • Replace generic tools with constrained alternatives: load_document that validates document URLs is better than a generic fetch_url.
  • Description quality drives adoption — a well-described MCP tool wins over a built-in like Grep.

tool_choice

5. MCP tool and resource design D2

What it is

MCP servers expose both tools (actions the agent invokes) and resources (content catalogs the agent can browse). Designing which pieces are which is a key exam concept.

RvT — Resources = read-only content · Tools = actions with side effects

Key facts

  • Resources reduce exploratory tool calls — give the agent a directory listing (issues, schemas, docs) up front instead of making it grep to discover.
  • Tools do things (create ticket, run query, send email). Resources let you look at things (list open tickets, browse schema).
  • Community MCP servers exist for standard integrations (Jira, GitHub, Slack) — prefer them over custom implementations. Reserve custom MCP servers for team-specific workflows.
  • Tool description quality drives adoption. Weak descriptions mean Claude falls back to built-in tools like Grep.

6. MCP server configuration D2

What it is

How and where you declare MCP servers so Claude can connect to them.

MCP-PES — Project scope · Env-var expansion · Simultaneous multi-server access

Where to declare them

FileScopeUse for
.mcp.json (repo root)ProjectTeam-shared tools, version-controlled
~/.claude.jsonUserPersonal or experimental servers

Environment variable expansion

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {"GITHUB_TOKEN": "${GITHUB_TOKEN}"}
    }
  }
}

Claude Code expands ${VAR} at connection time from your local environment — this keeps secrets out of version control.

Key facts

  • All configured servers' tools are discovered at connection time and available simultaneously.
  • Tool name collisions across servers must be resolved by the server (namespace).
  • Expose content catalogs (issue lists, doc hierarchies, DB schemas) as MCP resources so the agent doesn't need exploratory tool calls.

7. Error handling and propagation D2

What it is

Structured error responses that let the agent decide how to recover — retry, ask user to fix input, explain policy, or escalate.

TVBP — Transient (retry) · Validation (fix input) · Business (explain policy) · Permission (escalate)

Structured error response

{
  "isError": true,
  "errorCategory": "business",
  "isRetryable": false,
  "errorCode": "REFUND_LIMIT_EXCEEDED",
  "developerMessage": "Amount $650 exceeds $500 policy limit",
  "humanMessage": "Refunds over $500 require manager approval."
}

Anti-pattern

Uniform errors like {"error": "Operation failed"} prevent the agent from choosing an appropriate recovery. It can't tell whether to retry, ask the user, or escalate.

The empty-result vs access-failure distinction

Empty result = successful query, no matches. Return {"isError": false, "results": [], "querySuccessful": true}. This is NOT an error — the query worked.
Access failure = the query couldn't complete. Return an isError: true response with the appropriate category.

Local recovery principle

Subagents handle transient failures LOCALLY (retry once, degrade gracefully). Only propagate errors they cannot resolve — and include partial results when possible so the coordinator can decide what to do.

8. Escalation decision-making D2

What it is

How the agent decides when to hand off to a human — and how to make that decision reliable.

EES — Explicit criteria · Empirical customer preference · Surface policy gaps

Key facts

  • Explicit escalation criteria in the system prompt with few-shot examples of "when to escalate" outperform vague guidance.
  • LLM self-reported confidence is poorly calibrated — the model is often confidently wrong on hard cases. Don't rely on it as an escalation signal.
  • Honor customer preferences: if the customer says "get me a human", escalate immediately regardless of criteria.
  • Log escalations so recurring patterns surface policy gaps back to the team.

Exam scenario (page 26)

An agent misclassifies which support cases need escalation. Options include: (a) add explicit criteria + few-shot examples, (b) trust the model's self-confidence, (c) train a separate classifier, (d) use sentiment analysis. Answer: (a). This is the proportionate first response — try prompt improvements before adding infrastructure. (b) fails because LLM self-confidence is unreliable, (c) is over-engineered, (d) solves a different problem.

Domain 3 — Claude Code Configuration & Workflows (20%)

9. CLAUDE.md configuration D3

What it is

The hierarchical configuration system that tells Claude how to behave in a given project.

UPD — User level · Project level · Directory level

The hierarchy

ScopeLocationShared with team?
User~/.claude/CLAUDE.mdNo — personal only
Project.claude/CLAUDE.md or root CLAUDE.mdYes, via version control
DirectorySubfolder CLAUDE.mdYes, applies within that subtree

The classic diagnosis question

A new team member isn't receiving an instruction you added. Most likely cause: you placed it in ~/.claude/CLAUDE.md (user-scoped, not shared via VCS). Move it to .claude/CLAUDE.md or the root CLAUDE.md.

Modularity

  • Use @import syntax to reference external files: @packages/api/standards.md
  • Split large CLAUDE.md into topic files in .claude/rules/: testing.md, api-conventions.md, deployment.md
  • Use /memory command to inspect which memory files are actually loaded.

Path-scoped rules (see also Topic 3.3)

---
paths: ["**/*.test.tsx"]
---
# Test file conventions
- Use React Testing Library, not Enzyme
- Wrap async tests with `waitFor`

This rule loads ONLY when editing matching files, saving tokens. Prefer path-scoped rules over subdirectory CLAUDE.md when the convention spans multiple directories.

10. Custom commands and skills D3

What it is

Reusable, versioned workflows you can invoke on demand: slash commands (parameterised prompts) and skills (larger workflows with tool restrictions).

SKILL — context:fork · allowed-tools · argument-hint

Commands vs skills

CommandsSkills
Location (project).claude/commands/*.md.claude/skills/*/SKILL.md
Location (user)~/.claude/commands/~/.claude/skills/
Invocation/command-nameAuto or explicit skill invocation
Best forShort parameterised promptsLarger workflows with tool restrictions

Skill frontmatter fields

---
name: analyze-codebase
description: Deep-analyze a codebase and produce a summary
context: fork
allowed-tools: [Read, Grep, Glob]
argument-hint: <path-to-analyze>
---

Why context: fork

context: fork runs the skill in an isolated sub-agent context. Its output stays out of the main conversation. Perfect for skills that produce verbose output (codebase analysis) or exploratory context (brainstorming) — you don't want that flooding your main session.

Personal customization

Want a personal variant of a team skill? Create it in ~/.claude/skills/ with a different name. It won't affect teammates but is available in your sessions.

Skills vs CLAUDE.md — when to use which

  • CLAUDE.md = always-loaded universal standards.
  • Skills = on-demand invocation for task-specific workflows.

11. Plan mode vs direct execution D3

What it is

Deciding whether to enter Claude Code's Plan Mode (explore + design first) or execute changes directly.

PLAN — Parallel valid approaches · Large-scale · Ambiguous · Novel domain

When plan mode wins

  • Architectural implications (microservice restructuring, library migrations affecting 45+ files)
  • Multiple valid approaches — you need to compare before committing
  • Multi-file modifications where the diff shape is not yet clear
  • Unfamiliar area of the codebase — you need to explore first

When direct execution wins

  • Single-file bug fix with clear stack trace
  • Adding a validation conditional
  • Well-scoped rename or refactor
  • Any change where the diff shape is obvious before you start

Combining the two

Common pattern: use plan mode for investigation, then exit plan mode and use direct execution for implementation. The Explore subagent can isolate verbose discovery output so it doesn't consume your main context window.

12. Iterative refinement D3

What it is

Techniques for progressive improvement of a solution — bringing Claude to the right answer through examples, tests, or dialogue rather than one-shot prompting.

TIC — Tests-first · I/O examples · Conversation-interview

Techniques (guide page 15)

  • Concrete I/O examples — 2-3 pairs of (input, expected output) beat prose descriptions when prose is being interpreted inconsistently.
  • Test-driven iteration — write test suites first (covering happy path, edge cases, performance), then iterate by sharing test failures. Claude fixes based on failing tests rather than your descriptions.
  • Interview pattern — have Claude ask you questions before implementing (surfaces considerations you hadn't anticipated — cache invalidation, failure modes, etc.).

Together vs sequential

Together (single message) — when the issues interact. Claude needs to see the full picture so its fix for A doesn't break B.
Sequentially (across messages) — when the issues are independent. Fixing one shouldn't require reasoning about the other.

Domain 4 — Prompt Engineering & Structured Output (20%)

13. Structured output via tool_use D4

What it is

Getting Claude to return JSON conforming to a schema by defining that schema as a tool's input_schema and forcing the tool via tool_choice.

SNT — Schema-first · Nullable fields · Tool_choice pin

The pattern

tool = {
    "name": "extract_customer_info",
    "input_schema": {
        "type": "object",
        "properties": {
            "name": {"type": "string"},
            "email": {"type": ["string", "null"]},  # nullable
            "tier": {"type": "string", "enum": ["premium", "standard", "other"]},
            "tier_detail": {"type": ["string", "null"]}  # required when tier == "other"
        },
        "required": ["name", "email", "tier"]
    }
}
resp = client.messages.create(
    ...,
    tools=[tool],
    tool_choice={"type": "tool", "name": "extract_customer_info"}  # forced
)

Nullable fields

Mark uncertain fields as nullable. Claude will return null when the source doesn't have the info — this prevents hallucination. Without nullable, Claude will invent a plausible value.

Enum + "other" pattern

For classifications with an open-ended fallback: give an enum with a literal "other" option, plus a nullable detail field. Validate that when the classification is "other", the detail field is populated.

14. Few-shot prompting D4

What it is

Providing 2-5 example (input, output) pairs in the prompt to guide the model on ambiguous or edge cases.

FEW — Format consistency · Edge cases first · Wrong-answer examples included

Key principles

  • Target ambiguous cases — few-shot's job is to disambiguate. Easy cases don't need examples.
  • Format consistency — same shape across all examples (labels, punctuation, casing).
  • Include false-positive examples — show the model what NOT to classify as X, not just what IS X. This reduces false positives.
  • 3-5 examples is the usual sweet spot. Diminishing returns beyond that.

15. Batch processing D4

What it is

Using the Message Batches API for high-volume, non-urgent workloads. 50% discount vs sync, up to 24h latency.

BATCH — Best for non-urgent · Amount 50% off · Tool use single-turn only · Custom_id correlate · Handle per-item failures

When it fits

Use batches whenDo NOT use batches when
Overnight extraction from 10,000 docsInteractive customer support
Bulk document classificationMulti-turn tool loops (batches are single-turn)
Nightly report generationAnything user-facing with a latency SLA

Correlation via custom_id

Each batch request carries a custom_id. Responses come back with the same id so you know which request each response answers. Retry only the failed items — reuse the id with a suffix like id + "__retry".

A batch request with params: {} is invalid. When resubmitting failed items, carry the original params forward — don't emit an empty params dict.

Domain 5 — Context Management & Reliability (15%)

16. Context window optimization D5

What it is

Techniques to fit more useful information into the context window and to keep the model's recall of critical facts high.

TRIM — Trim verbose outputs · Reorder for position · Important facts extract · Middle-of-context loss

Techniques

  • Trim verbose tool outputs — summarise before appending to messages. A 50KB grep result usually needs only the 5 relevant lines.
  • Structured fact extraction — pull key facts out of long docs into a compact structured summary; feed the summary instead of the raw doc.
  • Position-aware input ordering — critical facts at START and END of prompt. Middle-of-context recall is measurably weaker ("lost in the middle").
  • Scratchpad files — persist working state to disk when it outgrows the context window.

The "lost in the middle" effect

Studies show recall drops for content in the middle third of a long prompt. If you're stuffing a 100KB document in and asking a question about it, put the critical instructions BEFORE and AFTER the document — not just before.

17. Human review workflows D5

What it is

Routing uncertain LLM outputs to humans for review — with the confidence threshold based on real measured error rates, not the model's self-report.

CALIBRATE — Confidence calibrated · Accuracy segmented · Labeled ground truth · Implementation via sampling · Boundary threshold routes to human

Core insight

LLM self-reported confidence is poorly calibrated. A model can be 0.95 confident and wrong. You can't use raw model confidence as the routing signal — it needs calibration.

The calibration workflow

  1. Build a labeled validation set (ground truth for 100-1000 documents).
  2. Run the extraction on the validation set; record model confidence AND correctness.
  3. Compute: at each confidence threshold X, what is the error rate?
  4. Set threshold based on your business tolerance for errors.
  5. Stratified sampling: measure error rate broken down by document type AND by field — accuracy is not uniform.

18. Information provenance D5

What it is

Tracking WHERE each claim came from, WHEN, and how reliable it is. Essential for research and analysis outputs where the user needs to verify.

CLASP — Claim · Linked source · As-of date · Status · Partial coverage reported

What every finding should carry

  • Claim — the specific statement being made
  • Source — URL or document reference (not "a study" — an actual link)
  • As-of date — when was this true? A 2019 fact may be outdated in 2026.
  • Status — WELL_ESTABLISHED (multiple corroborating sources) / CONTESTED (sources disagree) / UNVERIFIED (single-source)

Conflict annotation

When two sources disagree, DO NOT silently pick one. Annotate the conflict, cite both sources with dates, mark the claim CONTESTED, and let the user decide.

Coverage gaps

Report what you DIDN'T find, not just what you did. "I found data for 2020-2024 but no 2025 figures were available" is more useful than silently omitting 2025.
